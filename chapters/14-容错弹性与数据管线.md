## 14.0 为什么需要这一层

阶段 13 解决了“卡怎么分给作业”，假设的是作业一旦拿到卡就能安稳跑完。**但在千卡规模上，这个假设是错的。** 故障不是意外，是常态：单卡、单网卡、单根光纤的平均无故障时间（MTBF）哪怕长达一年，把上千个这样的部件串在一个**必须全员在线**的同步训练作业里，期望故障间隔就被压到了**几小时一次**——大模型训练日志里动辄上百次重启，就是这么来的。

这就逼出一个单作业视角（阶段 1–12）从不考虑的问题：**作业一定会在跑到一半时挂掉，怎么办？** 一个没有容错的训练脚本，挂一次就从头再来；在故障频发的规模上，它可能**永远训不完**——还没跑到下一个 checkpoint 就又挂了，有效进度（goodput）趋近于零。让大模型训练在可接受成本内真正“训得出来”，靠的不是单步更快，而是**这一层可靠性工程**：

1. **故障检测**：卡死了、某个 rank 拖后腿、loss 变 NaN、甚至算错了却不报错（静默数据损坏）——怎么**及时发现并定位**，而不是让整个作业陪着一个坏 rank 空转几小时（§14.3）。
2. **弹性与恢复**：发现故障后，怎么换掉坏节点、重组通信组、从最近的 checkpoint **自动爬起来续跑**，而不是等人来重启（§14.4）。
3. **Checkpoint 工程**：存盘本身会打断训练、占带宽。怎么**异步、分层**地存，让“丢失的进度”和“存盘开销”都尽量小（§14.5，深化阶段 7 §7.8 只讲过的格式）。

可靠性之外，还有两条**喂养**作业的管线，规模一大同样会成为瓶颈：

4. **训练数据管线**：TB 级数据集怎么分片、流式读进来，不让 dataloader 拖慢昂贵的 GPU（§14.6）。
5. **权重加载与分发**：几十上百 GB 的权重，怎么从存储快速灌进显存——这既是训练 resume 的一步，也是推理冷启动（阶段 13 §13.7.1 的 scale-to-zero）的命门（§14.7）。

最后用一套 cookbook 把检测和恢复串成实战（§14.8）。本章横跨两种写法：**§14.2–§14.7 是原理**（A 类），**§14.8 是排障 cookbook**（D 类），小节标题已标注类型。

读完之后你应当能：

1. 用 MTBF 估算一个千卡作业多久挂一次，说清为什么没有容错就训不完（§14.2）；
2. 区分 hang / straggler / NaN / SDC 四类故障，各用什么手段检测（§14.3）；
3. 解释 torch elastic 的 rendezvous 怎么让作业在节点变动后自动重组续跑（§14.4）；
4. 设计 checkpoint 策略：间隔怎么定、为什么要异步、怎么并行加载（§14.5）；
5. 搭一条不拖慢 GPU 的数据管线、并把权重冷启动从分钟级压到秒级（§14.6–§14.7）。

## 14.1 核心概念与术语

本章术语分两摊：可靠性（检测 / 恢复 / 存盘）和管线（数据 / 权重）。

| 术语 | 全称 / 中文 | 一句话 |
|---|---|---|
| MTBF | Mean Time Between Failures | 平均无故障时间；部件越多、同步越强，作业级 MTBF 越短 |
| goodput | 有效吞吐 | 真正推进训练的算力占比（扣掉丢失进度 + 停摆），容错的最终目标 |
| checkpoint | 检查点 | 把训练状态（参数 + 优化器 + step + RNG）存盘，故障后回到这一步 |
| 异步 checkpoint | async checkpoint | 先把状态拷到 CPU、后台慢慢落盘，训练几乎不停（§14.5） |
| straggler | 拖后腿 rank | 某个 rank 异常变慢，同步训练被它拖住（全员等最慢的） |
| hang | 卡死 | 作业不前进也不报错，常因 collective mismatch / 死锁 |
| NCCL timeout / watchdog | — | collective 超时机制，把“静默 hang”变成可捕获的报错（阶段 11 §11.6） |
| SDC | Silent Data Corruption | 硬件算错却不报错，结果悄悄变坏，最隐蔽的故障 |
| loss spike / NaN | — | loss 突然飙升或变 NaN，数值发散的信号 |
| torch elastic / torchrun | — | PyTorch 的弹性启动器，节点变动后靠 rendezvous 重组继续 |
| rendezvous | 集合点 | 弹性训练里各 worker “重新点名、组队”的机制 |
| WebDataset / StreamingDataset | — | 把数据集打成分片、从远端流式读取的格式，避免本地全量落盘 |
| weight streaming | — | 边加载边用权重，把模型冷启动从分钟压到秒（tensorizer 等） |
| 对象存储 | object storage | S3 / GCS 等，存权威 checkpoint 和数据集的“最底层、最大、最慢” |

> 阅读本章的心智准备：**容错的本质是“用冗余换可用性”**——checkpoint 是时间维度的冗余（存历史状态）、弹性是资源维度的冗余（备节点顶上）、检测是把“静默变坏”变成“显式报错”。三者合起来，让一个由会坏的部件组成的系统，整体上能持续推进。下一节先把这套回路画清楚。

## 14.2 全局图：故障面与可靠性回路（A 类）

类型 A 的起手式：先把“为什么需要这一层”用一张图和一个估算钉死，再逐项展开。

![大规模训练的可靠性回路：故障 → checkpoint → 检测 → 弹性重启 → 续跑](../svg/32-training-reliability-loop.svg)

### 14.2.1 先算一笔账：千卡作业多久挂一次

为什么单作业视角可以忽略故障、千卡视角不能？做个估算就清楚了。假设单个 GPU（连同它依赖的网卡、链路）的 MTBF 是相当乐观的**一年**（约 8760 小时）。一个同步训练作业要求**所有**卡同时在线——任何一张挂了，整个作业就得停。那么 $N$ 张卡的作业，期望故障间隔约为：

$$T_\text{fail} \approx \frac{\text{单卡 MTBF}}{N}$$

代入 $N=1024$：$8760 / 1024 \approx 8.6$ 小时挂一次。$N=10000$ 时不到 1 小时一次。这还只算了 GPU；加上交换机、光模块、电源、软件 bug，真实大集群的作业级故障间隔常以**小时**计。

这个估算解释了一切：**在这个尺度上，“一次训练跑到底”根本不存在。** 一个没有容错、挂了就从头再来的作业，如果跑满一遍要 100 小时、而每 8 小时挂一次，它永远跑不到终点——每次都没到下一个里程碑就又倒了。**容错不是优化项，是大模型训练能不能完成的前提。**

### 14.2.2 可靠性回路：丢失进度 + 停摆，两个都要压

既然故障必然发生，目标就从“不出故障”转成“**出故障后快速、少损失地恢复**”。这就是上图的回路：作业正常跑 step，**周期性存 checkpoint**；某一刻故障砸下来（上图红 ✗）；系统**检测**到、**弹性重组**、**从最近的 checkpoint 加载**、**续跑**。

衡量这套回路好不好，看 goodput——有效推进训练的时间占比。一次故障浪费的时间分两块（对应上图两段标注）：

- **丢失的进度** = 距上次 checkpoint 的时间。故障发生时，上个 checkpoint 之后跑的 step 全白干、得重来。**它由 checkpoint 间隔决定**：存得越密，丢得越少。
- **停摆时间** = 检测 + 重启 + 加载 checkpoint。这段作业完全不产出。**它由检测速度、调度恢复、加载速度决定**。

$$\text{goodput} \approx \frac{\text{有效训练时间}}{\text{有效训练时间} + \text{丢失进度} + \text{停摆时间}}$$

两块此消彼长，要一起压：**checkpoint 存得越密，单次丢的进度越少，但存盘本身的开销越大、越频繁打断训练**——所以不能无脑加密。最优间隔是在“丢失进度”和“存盘开销”之间取平衡（经典结论：最优间隔约正比于 $\sqrt{\text{MTBF} \times \text{单次存盘耗时}}$）。而**异步 checkpoint**（§14.5）能把“单次存盘耗时”几乎压没，从而允许存得更密、丢得更少——这就是它的价值。停摆时间那一块，则靠快速检测（§14.3）、弹性重组（§14.4）、并行加载（§14.5/§14.7）来压。

### 14.2.3 一套存储层级，贯穿三件事

最后看上图最下面那条带：checkpoint 往哪存、数据从哪来、权重从哪加载——**三件事共用同一套存储层级**（HBM → CPU 内存 → 本地 NVMe → 对象存储，回阶段 0 §0.2.2 的内存层级、阶段 5 §5.8 的 KV offload 同款思路）。规律一致：**越往下容量越大、越慢**。

- **checkpoint** 往下刷：先异步拷到 CPU、再后台落到 NVMe / 对象存储（§14.5）；
- **训练数据** 往上流：从对象存储 / NVMe 流式读进来喂 GPU（§14.6）；
- **权重** 往上加载：从对象存储把模型灌进 HBM（§14.7）。

三者都在和这套层级的**带宽**较劲——这也是为什么本章后半（§14.5–§14.7）反复回到“怎么把搬运藏起来 / 加速”。

> 一句话：**千卡作业每几小时挂一次（$T_\text{fail}\approx$ 单卡 MTBF / N），所以容错是前提而非优化。** 可靠性回路 = 周期 checkpoint + 检测 + 弹性重启 + 从 ckpt 续跑；要同时压“丢失进度”（靠 checkpoint 间隔 + 异步存盘）和“停摆时间”（靠快检测 + 弹性 + 并行加载）。checkpoint / 数据 / 权重三件事共用一套越往下越慢的存储层级，后面各节都在和它的带宽较劲。

## 14.3 故障检测：hang / straggler / NaN / SDC（A 类）

可靠性回路的第一环是**检测**。它难就难在：故障里最麻烦的几种**不会报错**——作业看着还在跑、GPU 灯还亮着，但要么悄悄卡住、要么悄悄变慢、要么悄悄算错。一个坏 rank 能拖着另外 1023 张卡空转好几个小时，直到有人发现。检测的全部功夫，就是**把这些"静默"尽快变成"显式信号"**。

### 14.3.1 检测的总思路：四类故障，四种信号

四类故障各有"现形"的办法，对应四种检测信号：

| 故障 | 表现 | 会报错吗 | 检测信号 |
|---|---|---|---|
| **hang**（卡死） | 不前进也不退出 | 否 | 超时（NCCL watchdog）+ 进度停滞 |
| **straggler**（拖后腿） | 某 rank 异常慢 | 否 | 统计离群（per-rank 时间分布） |
| **NaN / loss spike** | loss 飙升或变 NaN | 半（能算出但不崩） | 阈值监控（loss / grad norm） |
| **SDC**（静默损坏） | 结果悄悄算错 | **否** | 主动校验（一致性 / 自检） |

记住这张表的右列：**只有数值异常算是"半显式"，其余三类都得靠监控主动揪出**。下面逐个看。

### 14.3.2 hang：最常见的根因是 collective mismatch

同步训练里所有 rank 必须步调一致地调用每一个集合通信（AllReduce / AllGather…）。一旦某个 rank 因为分支逻辑、数据形状不一致等原因**少调或多调了一次 collective**，其余 rank 就会在对应的通信处**永远等它**——整个作业卡死，没有任何报错。这就是 collective mismatch，是大规模训练 hang 的头号原因（回阶段 11 §11.6）。

把它变显式靠两件套：

```bash
# NCCL watchdog：collective 超时就报错并 abort，而不是无限等
export TORCH_NCCL_BLOCKING_WAIT=1          # 或 ASYNC_ERROR_HANDLING=1
export TORCH_NCCL_TIMEOUT=600              # 超时阈值（秒）
# Flight Recorder：环形记录每个 rank 最后若干 collective，hang 后 dump 出来比对
export TORCH_NCCL_TRACE_BUFFER_SIZE=2000
```

- **watchdog** 把"无限 hang"变成"超时异常 + abort"——至少作业会**主动倒下**，触发后面的恢复（§14.4），而不是哑等几小时；
- **flight recorder** 在 hang 后 dump 出每个 rank 的最后几个 collective，**一眼看出哪个 rank 卡在哪、谁和谁对不上**——定位 mismatch 的利器。

此外还要一层**进度心跳**：训练循环每步更新一个 step 计数器 / 时间戳，外部监控发现它一段时间不涨就报警——这能兜住 watchdog 也抓不到的非通信类卡死（如某 rank 死在数据加载、磁盘 IO）。

### 14.3.3 straggler：没人报错，得靠统计揪出

straggler 是"没死但慢"：一张卡因为降频、ECC 持续修正、风扇/温度问题、网络劣化、或数据加载慢，比别人慢一截。同步训练**全员等最慢的那个**——一个 straggler 能把整批的吞吐拉到它的水平，而且**完全不报错**。

检测只能靠**统计离群**：采集 per-rank 的指标分布，找异常值。

- **每 rank 的 step 耗时 / collective 等待时间**：正常时各 rank 接近，某个 rank 的"计算时间"突出、或其余 rank 的"通信 wait"突然变长（都在等它），就锁定了它；
- **per-rank 的 SM 利用率 / 时钟 / 温度**（DCGM 采集）：辅助确认是硬件降频还是别的；
- 实践里常设一个阈值（如慢于中位数 1.5×）自动标记 straggler，重的直接当故障处理——把它的节点换掉（§14.4），好过让全员陪跑。

### 14.3.4 NaN / loss spike：监控曲线 + 回滚

数值异常是"半显式"的：loss 突然飙升或变 NaN，模型在发散。根因可能是学习率过大、一个坏 batch、混合精度（BF16/FP8）下的溢出、或梯度爆炸。它能被算出来、但训练不会自己崩，所以要**主动盯**：

- 实时监控 **loss 和 grad norm** 曲线，设阈值报警；在 loss / 梯度上跑 `torch.isnan` / `isinf` 检查；
- 触发后的标准动作：**回滚到上一个 checkpoint**，跳过或重排触发问题的 batch、调低学习率后续跑（所以 checkpoint 还兼当"数值事故的存档点"）；
- 留住触发前的 ckpt + 那批数据，便于复现定位——很多 spike 是特定脏数据引起的。

### 14.3.5 SDC：最隐蔽的，只能主动校验

最难对付的是 **SDC**（Silent Data Corruption，静默数据损坏）：硬件算错了却**不报任何错**——某个计算单元偶发出错、一个 bit 悄悄翻转，结果就这么带着错误往下走。它不 hang、不报 ECC、loss 也未必看得出异常，但模型在悄悄变坏。大厂的大规模训练实践里，SDC 已被确认是真实且影响显著的故障源。

因为没有任何被动信号，只能**主动校验**：

- **一致性校验**：数据并行的多个副本，相同输入本应得到相同结果——跨副本比对，不一致就说明有 rank 算错了；
- **确定性自检**：周期性用固定输入跑一小段、和已知正确结果（或自身重算）比对；
- **厂商诊断**：DCGM diagnostics、NVIDIA 的 SDC 检测工具，定期体检揪出可疑卡，从池子里剔除（呼应阶段 13 的节点健康检查）。

> 一句话：**检测的本质是把"静默"变"显式"——hang 靠 NCCL watchdog + flight recorder（揪 collective mismatch）、straggler 靠 per-rank 统计离群、NaN 靠 loss/grad 阈值 + 回滚、SDC 靠主动一致性校验。** 四类里只有数值异常半显式，其余都得监控主动出击；检测越快，§14.2 的"停摆时间"越短。检测到之后，怎么爬起来续跑是下一节。

## 14.4 弹性与恢复：torch elastic 与 rendezvous（A 类）

检测把故障变成了显式信号、作业主动倒下。恢复要回答的是：**没有人盯着的半夜，作业怎么自己爬起来续跑？** 这一节讲"爬起来"的进程层机制；"续跑用的状态"（checkpoint）留到 §14.5。

### 14.4.1 恢复 = 把进程拉回来 + 把状态 load 回来

一次恢复拆成两件正交的事：

1. **进程层**：坏节点踢掉、（可能）换上备用节点、把存活的 worker 重新组成一个通信组——这是 **torch elastic / torchrun** 干的（本节）；
2. **状态层**：新组起来的 worker 从**最近的 checkpoint** 加载参数 / 优化器 / step / RNG / 数据位置，回到故障前那一步——这是 checkpoint 的事（§14.5）。

两件事缺一不可：只重组进程不 load 状态，等于从头训；只有 checkpoint 没有自动重组，还得人来重启。容错系统要的是**两者自动衔接**。

### 14.4.2 torch elastic：靠 rendezvous 自动重组

`torchrun`（PyTorch 的弹性启动器）的核心是 **rendezvous**（集合点）：每次 worker 启动或重启，它们先去一个约定的"集合点"**重新点名、组队**——商定这一轮有几个节点、各自的 rank、谁是 master。一旦某个 worker 失败，torchrun 检测到，就**重启整组 worker、触发一次新的 rendezvous**，组好后各 worker 从 checkpoint 恢复、继续训练。整个过程不需要人介入。

```bash
torchrun \
  --nnodes=14:16 \                 # 弹性范围：最少 14 节点也能跑，最多 16
  --nproc-per-node=8 \
  --max-restarts=10 \              # 自动重启上限（超了才真正放弃）
  --rdzv-backend=c10d \            # rendezvous 后端（c10d / etcd）
  --rdzv-endpoint=$HOST:29400 \    # 集合点地址
  --rdzv-id=pretrain-llama-0617 \  # 同一作业的 worker 用同一个 id 组队
  train.py
```

两个关键参数：

- **`--nnodes=MIN:MAX`** 是"弹性"的来源——给出一个**节点数范围**而非固定值。某些节点挂了、只要还剩 MIN 个，作业就能在新的规模上继续，不必死等全部到齐；
- **`--max-restarts`** 是自动重启的次数上限——在它之内，故障→重启→恢复全自动；超过才真正失败退出、留给人处理。

### 14.4.3 为什么失败要"整组重启"

一个反直觉但关键的点：torchrun 在任一 worker 失败时，会**杀掉并重启整组 worker**，而不是只补那一个。原因是同步训练的状态是**全局耦合**的——所有 rank 的参数、优化器、通信组是一个整体。单独重启一个 worker，它的通信组、step、随机状态都和其余 rank 对不齐，没法直接插回去。所以最稳的做法是**整组回到上一个一致的 checkpoint**，从那里重新出发。这也呼应阶段 13 §13.7.2：**抢占 / 故障的处理粒度是 gang（整组），不是单个 worker**。

代价是：哪怕只挂一张卡，整个作业都要重启一次——这就是 §14.2 "停摆时间"的主要来源，也是为什么"快速重启 + 并行加载 checkpoint"（§14.5）这么重要。

### 14.4.4 弹性的真相：纯缩容很难，现实多是"替换 + 保形"

"弹性"听着像"挂几张卡就用剩下的接着跑"，但对 **3D 并行**作业这基本做不到。TP / PP / EP 的并行布局是**和卡数强绑定**的：TP=8 就需要 8 张卡组一个张量并行组，挂一张就不再是 8。真要"缩容续跑"，得在线**重排并行布局**（reshape）——这非常复杂，多数框架不支持。

所以现实里的"弹性"通常是两种较朴素的形态：

- **替换 + 保形**：坏节点踢掉，从**热备节点**（hot spare，预留几台空闲机，呼应阶段 13 调度）顶上一台，**保持原规模和并行布局**重启。这是大规模 3D 并行训练的主流做法——`--nnodes` 范围给的余量主要用来熬过"备机还没顶上"的瞬间，而不是真降规模。
- **纯缩容**：只有**纯数据并行**（DP）作业才比较容易——少几个 DP 副本，全局 batch 变小、调一下即可继续。一旦有 TP/PP，缩容就要 reshape，通常不划算。

实践要点：**预留热备节点**，让替换不必等云上现开新机（那要几分钟）；并把**全局 batch / 学习率**对节点数变化的依赖想清楚，避免恢复后训练动态偷偷变了。

> 一句话：**恢复 = torchrun 重组进程（rendezvous）+ 从 checkpoint load 状态，两者自动衔接。** 任一 worker 挂都整组重启（同步状态全局耦合，gang 粒度），所以停摆时间主要花在重启 + 加载。"弹性"对 3D 并行作业多是"替换坏节点 + 热备保形"，纯缩容只对 DP 作业现实——真要在线 reshape 并行布局太难。下一节看怎么把"丢失进度"和"加载耗时"一起压下去：checkpoint 工程。

## 14.5 Checkpoint 工程：异步、分层、并行加载（A 类）

阶段 7 §7.8 讲过 checkpoint 的**格式**（safetensors、分布式 ckpt）。本节讲它的**工程**：怎么存得勤、存得快、加载得快——直接决定 §14.2 那两块损失（丢失进度 + 停摆）能压到多小。

### 14.5.1 一个 checkpoint 要存的不只是权重

第一个常见错误是只存模型参数。要能**精确**恢复到故障前那一步，必须存全五样：

| 存什么 | 不存的后果 |
|---|---|
| 模型参数 | —（这个不会忘） |
| **优化器状态**（Adam 的 m / v） | 动量丢失，恢复后 loss 抖动、等于重新预热 |
| step / LR scheduler 状态 | 学习率调度错位 |
| RNG 状态 | dropout / 数据增强不可复现 |
| **dataloader 位置** | 数据重复训或被跳过（§14.6） |

注意优化器状态是大头：混合精度训练里，每个参数除了 BF16 权重，还要存 FP32 master + Adam 的 m 和 v，**checkpoint 体积常是模型权重的 3–4 倍**。所以 ckpt 又大又要存得勤——这正是下面所有优化的动机。

### 14.5.2 同步存盘的痛：训练停在那等盘

最朴素的写法是一个阻塞的 `torch.save`：训练循环走到存盘点，**整个作业停下**，等状态从 HBM 拷到 CPU、再写进磁盘 / 对象存储。大 ckpt 这一停就是几十秒甚至几分钟。

这就把 §14.2 的权衡逼到死角：**存得越勤、被打断越多**（开销大），于是不敢勤存，结果**单次故障丢的进度就大**。同步存盘让你在"勤存"和"少打断"之间二选一——而下面的异步存盘能把这个二选一直接拆掉。

### 14.5.3 异步 checkpoint：把落盘藏到训练后面

关键洞察：存盘其实是两步，快慢悬殊——

1. **快照到 CPU**：把状态从 GPU 显存拷到 CPU 内存。这步**必须短暂停一下**以保证一致性，但很快（一次 HBM→CPU 拷贝，可用 pinned memory + 独立 stream 进一步加速）；
2. **落盘**：从 CPU 内存写到 NVMe / 对象存储。这步**慢**，但和 GPU 计算无关。

异步 checkpoint 把第 1 步压到最小、第 2 步丢给后台线程，训练**立刻继续**。于是 GPU 的停顿从"整个落盘时间"缩到"一次显存快照"——**单次存盘开销几乎归零**，从而允许存得更勤、把"丢失进度"压到很小（呼应 §14.2 的最优间隔结论）：

```python
import torch.distributed.checkpoint as dcp

state = {
    "model": model.state_dict(),     # 参数（分片）
    "optim": optim.state_dict(),     # 优化器状态：Adam m/v，体积是权重的几倍
    "step":  step,
    "rng":   torch.get_rng_state(),
    "data":  dataloader.state_dict(),# 数据走到哪（§14.6）
}

fut = dcp.async_save(state, checkpoint_id=f"s3://ckpt/step-{step}")  # 主线程几乎不停
# ...继续跑后面的 step...
fut.result()   # 下次存盘前，确认上一次已落盘
```

### 14.5.4 分布式 + 分层：别让 rank0 当瓶颈

两个进一步的工程点：

- **分布式 checkpoint**（DCP，`torch.distributed.checkpoint`）：**每个 rank 只存自己持有的那一分片**、并行写，而不是把全部状态 gather 到 rank0 再存——后者既会让 rank0 OOM，又把并行写退化成单点串行。加载时 DCP 还能按**新的并行布局重新分片**（resharding），这正是"换了 GPU 数 / 并行度也能加载"的底气（呼应 §14.4.4）。
- **分层落点**（呼应 §14.2.3）：先写**本地 NVMe**（快，供近期故障快速恢复），后台再异步上传**对象存储**（权威、跨节点、灾备）。更激进的还有 **in-memory checkpoint**——把一份 ckpt 留在**另一台节点的 CPU 内存**里，单卡故障时直接从内存恢复（最快），再定期落盘兜底。

### 14.5.5 并行加载：压"停摆"的最后一段

恢复时的加载同样别走"rank0 读完再广播"的老路——那是单点瓶颈。让**每个 rank 并行**从存储读自己那一分片（DCP 原生支持）；权重本身的加载再叠加 streaming（§14.7）。这压的是 §14.2 "停摆时间"里的加载那一段。把异步存（压丢失进度）+ 分布式并行加载（压停摆）合起来，goodput 才能真正上去。

> 一句话：**checkpoint 要存全五样（参数 + 优化器 + step + RNG + 数据位置），体积是权重的几倍。** 异步存盘把"停下等落盘"压成"一次显存快照"，于是能勤存、少丢进度；分布式 ckpt 让各 rank 并行存/读自己那片、还支持 resharding；分层（NVMe + 对象存储 + in-memory）兼顾快与权威。存（压丢失进度）和加载（压停摆）两头都优化，§14.2 的 goodput 才提得起来。

## 14.6 训练数据管线：分片与流式读取（A 类）

可靠性之外，还有一条**喂养**作业的管线。GPU 是全集群最贵的资源，绝不能让它**空转等数据**——可一旦数据集到了 TB–PB 量级，"把数据准时喂上"本身就是个系统工程。

### 14.6.1 为什么数据管线会拖慢昂贵的 GPU

数据从存储到 GPU 要过三道坎：**量大**（装不下本地盘）、**预处理耗 CPU**（解码 / tokenize）、**随机读 IO 慢**。任何一道跟不上 GPU 的消耗速率，GPU 就周期性空转等下一个 batch——这是 §0.2.2 的 **overhead-bound**：算力和带宽都闲着，时间花在等数据上，MFU 直接掉。

目标因此很明确：**数据供给速率 ≥ GPU 消耗速率，且把数据 IO 藏到计算后面**。下面三件事分别解决"装不下""读得慢""恢复要对齐"。

### 14.6.2 流式读取：不落盘、边下边训

第一个办法是**别在本地存整个数据集**。把数据集打成**分片**（shard，常是 tar 包，每包几百 MB）存到对象存储，训练时**流式**拉取分片、边下边解边喂，本地只留正在用的那几片。两个代表实现：

```python
import webdataset as wds
ds = (
    wds.WebDataset("s3://data/shard-{000000..009999}.tar", shardshuffle=True)  # 分片级打乱
      .shuffle(10000)                 # shard 内 shuffle buffer：内存里局部洗牌
      .decode().to_tuple("txt").map(tokenize)
)
loader = wds.WebLoader(ds, num_workers=8, batch_size=64)   # 多 worker 预取
```

- **WebDataset**：tar 分片 + `IterableDataset`，顺序读 tar、流式解；
- **MosaicML StreamingDataset**：自定义分片格式，内置确定性 shuffle 与**断点续读**（下面 §14.6.4 要用）。

好处：本地盘不必装下整个数据集；多节点 / 多 worker 各拉不同分片，天然可扩展。

### 14.6.3 sharding 与 shuffle：既要随机又要顺序读

这里有个矛盾：训练要充分 **shuffle**（数据顺序有偏会害训练），但存储（对象存储 / 磁盘）偏爱**顺序大块读**（随机小读极慢）。两个要求看似冲突。

解法是**两级洗牌**：

1. **分片级 shuffle**：打乱分片的拉取顺序（上面 `shardshuffle=True`）——粗粒度随机，但每片内部仍是顺序大块 IO；
2. **shard 内 shuffle buffer**：在内存里维护一个缓冲区（上面 `.shuffle(10000)`），样本进来先入缓冲、再随机取出——细粒度局部洗牌。

两级合起来，**近似全局随机，又保持顺序大块读**。同时多个 DP rank 要**不重不漏**地瓜分分片（每个 rank 拿不同的片），否则数据重复或浪费。

### 14.6.4 可恢复的 dataloader：和 checkpoint 对齐

回 §14.5.1 那张表的最后一行：恢复时数据必须**从断点继续**。如果 dataloader 不可恢复，重启后要么从头读（已训过的数据重训）、要么乱跳（漏训）——**数据分布被悄悄改变，训练动态偷偷走样**，还很难发现。

所以 dataloader 要支持 `state_dict` / `load_state_dict`，记录"走到哪个 shard、shard 内偏移、shuffle 种子与 buffer 状态"，并随 checkpoint 一起存（§14.5）。StreamingDataset 这类库内置了确定性恢复；自己写 IterableDataset 则要**亲手维护这个位置状态**——这是最容易被忽略、又最隐蔽伤训练的一环。

### 14.6.5 藏起 IO：预取 + 多 worker + overlap

最后是把数据搬运**藏到计算后面**，让 GPU 永远有下一个 batch 待命：

- **多 worker 预取**：`DataLoader(num_workers, prefetch_factor)` 多进程并行准备后续 batch；
- **pinned memory + 异步 H2D**：`pin_memory=True`，用独立 stream 把 batch 拷到 GPU、overlap 到当前 step 的计算后面（回阶段 0 §0.3）；
- **怎么确认是数据 bound**：profiler 里看 GPU 是否周期性空转、或 dataloader 的等待时间占比（回阶段 11）。是的话，加 worker、换更快的存储、或把预处理提前离线做。

> 一句话：**数据管线的目标是"GPU 不等数据"。** 流式分片读（WebDataset / StreamingDataset）解决"装不下"；两级 shuffle（分片级 + buffer）在"要随机"和"要顺序大块读"之间取平衡；可恢复的 dataloader 必须随 checkpoint 一起存位置，否则恢复后数据分布走样；预取 + 多 worker + 异步 H2D 把 IO 藏到计算后面。数据 bound 和算力 bound 一样要先 profile 再对症（阶段 11）。

## 14.7 权重加载与分发：压掉冷启动（A 类）

最后一条管线是**权重**。它看着是小事——不就是把模型读进来吗？但在两个场景里，它是命门：训练 resume 时的"停摆"（§14.4），和推理 **scale-to-zero** 的冷启动（阶段 13 §13.7.1）。两者较的是同一个劲：**把几十上百 GB 权重从存储灌进显存**，越快越好。

### 14.7.1 冷启动慢在哪

加载一个大模型的朴素路径是**串行三步**：从对象存储下载到本地 → 反序列化到 CPU 内存 → 拷进 GPU 显存。一个 70B BF16 模型 140 GB，这条串行路走下来动辄**几十秒到几分钟**。慢在三处：

1. **读取带宽**：从对象存储拉 140 GB，受存储 / 网络带宽限制；
2. **反序列化开销**：格式不友好（如 pickle）会逐张量拷贝、带 Python 开销；
3. **串行**：rank0 读完再广播、或单线程顺序加载，没有并行。

`safetensors` 已经比 pickle 强（零拷贝 mmap、按需载入、还安全），但只解决了第 2 点的一部分。要真正压下冷启动，得对三处一起下手。

### 14.7.2 weight streaming：边拉边灌，下载与拷贝重叠

关键思路和 §14.5.3 的异步 checkpoint 同源：**别等"全部下载完"再开始用，把下载和 H2D 拷贝流水化重叠起来**。tensorizer（CoreWeave）、Run:ai Model Streamer 这类工具，**逐张量地从对象存储直接流进 GPU**，一边拉下一个张量、一边拷上一个，多个流并行——冷启动从分钟级压到**秒级**：

```python
# 朴素：下载 → 反序列化到 CPU → 拷到 GPU，三步串行（分钟级）
model.load_state_dict(torch.load("model.pt"))

# 流式：逐张量从对象存储直接灌进 GPU，下载与 H2D 重叠（秒级）
from tensorizer import TensorDeserializer
des = TensorDeserializer("s3://models/llama-70b.tensors", device="cuda")
des.load_into_module(model)
```

`safetensors` 的 **mmap** 也是一种轻量办法：内存映射文件，OS 按需把用到的页换进来，省掉"先整体读进内存"这一步。

### 14.7.3 并行与就近：每 rank 读自己那片 + 本地缓存

另外两招针对"串行"和"重复读"：

- **并行加载**：和 §14.5.5 一样，每个 rank **并行**从存储读自己持有的那一分片，不经 rank0 中转——多卡作业尤其关键，否则 rank0 的带宽成了全局瓶颈；
- **就近 / 预热**：把权重预先拉到**节点本地 NVMe 或 CPU page cache**，下次冷启动从本地读（快得多）；推理侧则常保留一个 **warm 副本**（阶段 13 §13.7.1 的 `minReplicas: 1`），用"留一点常驻"换"首请求不卡几十秒"。这又回到那个权衡：**省卡（scale-to-zero）和低冷启动延迟，必须显式取舍**。

### 14.7.4 同一问题的两副面孔

把训练和推理摆一起看，会发现是同一件事：

| | 训练 resume | 推理冷启动 |
|---|---|---|
| 加载什么 | 参数 + 优化器状态（§14.5.1） | 只要参数 |
| 触发 | 故障 / 抢占后续跑（§14.4） | scale-to-zero 后首请求（§13.7.1） |
| 怎么压 | 分布式并行加载（§14.5.5） | weight streaming + warm 副本 |
| 较的劲 | **存储 → HBM 带宽** | **存储 → HBM 带宽** |

两者都在和 §14.2.3 那条存储层级的**最后一跳带宽**较劲——这也正好收束本章反复出现的主线：**checkpoint 往下刷、数据和权重往上灌，全章都在想方设法把这套层级的搬运藏起来、并行化、或就近缓存。**

> 一句话：**冷启动慢在"把几十上百 GB 串行搬进显存"。** 三处下手：weight streaming（tensorizer / Model Streamer）把下载与 H2D 重叠、压到秒级；各 rank 并行读自己那片不经 rank0；本地缓存 / warm 副本避免重复读。训练 resume 和推理 scale-to-zero 是同一问题的两副面孔，都在和存储→HBM 带宽较劲——这也是本章存储层级主线的收尾。

## 14.8 端到端排障实战（D 类 cookbook）

前面 §14.3–§14.7 是原理。这一节切到工具书写法：每个常见故障一格，**场景 → 命令 → 判读 → 下一步**，把检测手段串成实战。

### 14.8.1 作业卡住、step 不涨：定位 collective mismatch

**场景**：step 计数器一段时间不动，GPU 利用率卡在 100%（在等通信）或 0%，没有任何报错——典型 hang（§14.3.2）。

**命令**：先确认 watchdog 和 flight recorder 已开（§14.3.2 的环境变量）。等 watchdog 超时 abort 后看它 dump 的 trace；想立刻看各 rank 卡在哪，用 py-spy 抓栈：

```bash
# 逐个 rank dump Python 调用栈，看卡在哪一行
py-spy dump --pid <rank_pid>
# watchdog 超时后，flight recorder 会把各 rank 最后的 collective 记录 dump 出来
```

**判读**：flight recorder 若显示 `rank 5` 的最后一个 collective 是 `AllReduce#1234`、而其余 rank 已进到 `AllGather#1235`——**某个 rank 比别人少调（或多调）了一次集合通信**，就是 collective mismatch。py-spy 栈则会显示多数 rank 卡在 `all_gather`、个别 rank 卡在别处。

**下一步**：查那个掉队 rank 的代码分支——多半是某个条件（batch 里有无某字段、序列长度、loss 是否跳过）让它走了不同路径、少调了一次通信。对齐分支即可。`--max-restarts`（§14.4.2）只能兜住偶发，逻辑型 mismatch 必须改代码。

### 14.8.2 吞吐莫名掉一截：揪出 straggler

**场景**：MFU 从 50% 滑到 35%，没报错、loss 正常，就是变慢（§14.3.3）。

**命令**：采集 per-rank 的 step 耗时分布，并看每张卡的时钟 / 温度：

```bash
dcgmi dmon -e 1001,1002,150,203    # 每 GPU 的 SM 利用率 / 时钟 / 温度
# 或在训练里打印各 rank 的 step 时间，找离群值
```

**判读**：若 `rank 23` 的 step 时间是中位数的 1.6×、且 DCGM 显示这张卡**时钟降频、温度偏高**——硬件降频型 straggler。同步训练全员等它，于是整体吞吐被拉到它的水平。

**下一步**：把该节点 cordon 出去（阶段 13 §13.7），用热备节点顶上、整组重启（§14.4.4），坏卡报修。**别让一张慢卡陪跑全场。**

### 14.8.3 loss 变 NaN：定位触发点并回滚

**场景**：loss 突然变 NaN 或飙升，训练发散（§14.3.4）。

**命令 / 判读**：看 loss 和 grad norm 曲线（W&B / TensorBoard），定位 spike 的 step：

- 若 **grad norm 在 step N 先飙到很大再 NaN**——梯度爆炸，可能 bad batch 或学习率过大；
- 记下该 step 对应的数据分片，便于复现。

**下一步**：**回滚到 step N 之前的 checkpoint**（§14.5 的存档点正好派上用场），然后三选一或组合：加 / 收紧 grad clip、调低学习率、检查并跳过那批脏数据。改完从该 ckpt 续跑。

### 14.8.4 恢复后训练走样：查 checkpoint 完整性与数据位置

**场景**：从 checkpoint 恢复后，loss 比挂之前高一截、或明显抖动——恢复"成功"了，但训练动态变了。

**判读**：几乎都是 §14.5.1 那五样**没存全**。对照排查：

```bash
# 看 checkpoint 里到底存了哪些 key
python -c "import torch.distributed.checkpoint as dcp; print(dcp.load(...).keys())"
```

- 缺 **optim**（优化器状态）→ Adam 动量丢失，恢复后像重新预热，loss 抖；
- 缺 **data**（dataloader 位置，§14.6.4）→ 数据重训或跳过，分布偷偷变了；
- 缺 **rng** → dropout / 数据增强不可复现。

**下一步**：把 checkpoint 补全五样（§14.5.1）、换用可恢复的 dataloader（§14.6.4），重新验证"挂掉再恢复，loss 曲线应当无缝接上"。

> 一句话（cookbook 速记）：**step 不涨 → flight recorder + py-spy 查 collective mismatch；变慢不报错 → DCGM 揪降频 straggler；loss NaN → 看 grad norm 定位、回滚 ckpt 降 lr；恢复后走样 → 查 ckpt 是否存全五样 + 数据位置。** 四类故障的检测原理在 §14.3，这里是把它们落成"敲哪条命令、看哪个数、做什么动作"。

## 14.9 常见坑与 FAQ

1. **只存了模型权重、没存优化器状态**：恢复后 Adam 动量从零开始，loss 明显抖、像重新预热。checkpoint 要存全五样（§14.5.1）。
2. **dataloader 不可恢复**：重启后数据从头读（重训）或乱跳（漏训），数据分布悄悄变、还难发现。用可恢复 dataloader、把位置随 ckpt 存（§14.6.4）。
3. **用同步阻塞 `torch.save` 还存得很勤**：每次存盘整个作业停几十秒，吞吐被打断得很惨。改异步 checkpoint（§14.5.3），才敢勤存。
4. **checkpoint 间隔拍脑袋定**：太疏单次丢进度多、太密开销大。按 §14.2.2 的量级（约正比 $\sqrt{\text{MTBF} \times \text{单次存盘耗时}}$）估，异步存盘后可显著加密。
5. **hang 了干等、没开 watchdog**：作业可能哑等几小时没人知道。必开 NCCL watchdog + flight recorder（§14.3.2），把 hang 变成可捕获、可定位的事件。
6. **把整个 ckpt gather 到 rank0 再存**：rank0 容易 OOM，且单点串行写很慢。用分布式 checkpoint（DCP）各 rank 存自己那片（§14.5.4）。
7. **straggler 当没事**：一张降频的慢卡能把整批吞吐拉到它的水平。统计离群揪出、换节点（§14.3.3、§14.8.2）。
8. **以为 3D 并行作业能随便缩容**：TP/PP 布局和卡数强绑定，在线 reshape 极难。现实是"替换坏节点 + 热备保形"（§14.4.4）。
9. **scale-to-zero 后首请求超时**：冷启动慢在权重加载、不是调度。留 warm 副本或上 weight streaming（§14.7、阶段 13 §13.7.1）。
10. **只靠 loss 判健康，忽略 SDC**：硬件算错不报错、loss 未必看得出。要主动一致性校验 + DCGM 定期体检（§14.3.5）。

## 自测

1. **（口算）** 假设单卡 MTBF 约一年，一个 1024 卡的同步训练作业期望多久挂一次？为什么"没有容错就训不完"？
2. **（概念）** 一次故障浪费的时间分哪两块？各由什么决定、各靠什么手段压下去？
3. **（辨析）** hang / straggler / NaN / SDC 四类故障，哪些会报错、哪些不会？各用什么信号检测？
4. **（机制）** 一个 checkpoint 必须存哪五样？只存权重、漏了优化器状态和 dataloader 位置，恢复后各会发生什么？
5. **（机制）** 异步 checkpoint 把存盘拆成哪两步？为什么它能让你"存得更勤、丢得更少"？
6. **（概念）** 为什么任一 worker 失败要整组重启？对 3D 并行作业，"弹性"现实中是什么形态？
7. **（应用）** 训练数据怎么做到"既充分 shuffle、又对存储友好的顺序大块读"？
8. **（辨析）** 训练 resume 和推理 scale-to-zero 冷启动，共同的瓶颈是什么？各自怎么压？

<br>

**参考答案**

1. $T_\text{fail}\approx 8760\text{h} / 1024 \approx 8.6$ 小时一次。没有容错时，一次故障就从头再来；若跑完要 100 小时、每 8.6 小时挂一次，永远到不了终点——所以容错是**前提**而非优化。（§14.2.1）
2. **丢失的进度**（距上次 checkpoint，由 checkpoint 间隔决定，靠加密 + 异步存盘压）和**停摆时间**（检测 + 重启 + 加载，靠快检测 + 弹性重组 + 并行加载压）。goodput = 有效时间 / (有效 + 丢失 + 停摆)。（§14.2.2）
3. 只有 NaN / loss spike 半显式（能算出但不崩）；hang、straggler、SDC 都不报错。检测：hang 靠 NCCL watchdog + flight recorder、straggler 靠 per-rank 统计离群、NaN 靠 loss/grad 阈值、SDC 靠主动一致性校验。（§14.3）
4. 五样：参数 + 优化器状态 + step/LR + RNG + dataloader 位置。漏优化器状态 → 动量丢失、恢复后 loss 抖（像重新预热）；漏 dataloader 位置 → 数据重训或跳过、分布走样。（§14.5.1）
5. 两步：① 快照状态到 CPU（短暂停顿，很快）；② 后台从 CPU 落盘（慢，但和计算无关）。异步把第 1 步压到最小、第 2 步丢给后台，单次存盘开销几乎归零，于是能加密 checkpoint、把丢失进度压小。（§14.5.3）
6. 同步训练状态全局耦合，单独重启一个 worker 的通信组 / step / RNG 和其余对不齐，所以整组回到上一致 checkpoint 重启（gang 粒度）。3D 并行的"弹性"现实是"替换坏节点 + 热备保形"，纯缩容只对 DP 作业可行。（§14.4.3、§14.4.4）
7. 两级 shuffle：分片级打乱拉取顺序（粗粒度随机、每片内仍顺序大块读）+ shard 内 shuffle buffer（内存里局部洗牌）。近似全局随机，又保持顺序大块 IO。（§14.6.3）
8. 共同瓶颈是**存储 → HBM 的最后一跳带宽**（把几十上百 GB 灌进显存）。训练 resume 靠分布式并行加载（各 rank 读自己那片）；推理冷启动靠 weight streaming + warm 副本。（§14.7.4）

> 第 1 题把"为什么需要这一层"变成一个能算的数；第 2、5 题考可靠性回路的核心权衡（goodput 两块、异步存盘）；第 4 题是最常踩的坑（ckpt 没存全）。这几题答得出，容错这层就算拿下了。
